# 5.4.2. GET {e} (Service Provider Metadata)

Returns the Service Provider metadata document.

Schema: `../../3-operating-agentpass/3-schemas/service-metadata.schema.json`.

## Request

Client issues `GET {e}` resolved from Service Provider discovery.

Client SHOULD send `Accept: application/json`.

## Response

Service Provider MUST return JSON metadata with:

- `kind = "service"`
- `service.origin`
- `trust`
- `endpoints`

## Trust Declaration

`trust.accepted_authorities` lists trusted authoritative issuers.

`trust.accepted_federated_authorities` (optional) lists trusted federated issuers.

Service Provider trust configuration MUST include at least one of these lists.

Service Providers enforcing authoritative precedence MUST apply rules from:

- [`../../3-operating-agentpass/3.1-authoritative-vs-federated.md`](../../3-operating-agentpass/3.1-authoritative-vs-federated.md)
- [`../../3-operating-agentpass/3.5-federated-identity-providers.md`](../../3-operating-agentpass/3.5-federated-identity-providers.md)

## Endpoint Declaration

`endpoints.init`:

- absolute HTTPS URL for `POST` artifact redemption

`endpoints.initialize`:

- absolute HTTPS URL template for browser initialization
- template variable `{token}` identifies initialization token

`endpoints.resolve_authority`:

- absolute HTTPS URL for Runtime authority-resolution requests
- accepts delegating principal email and returns authority selection data, including `trust_mode`

## Consumer Behavior

Runtimes MUST use Service Provider `resolve_authority` endpoint (not direct authority discovery) to obtain authority selection for browser-session and bearer-token flows.

Service Providers remain the final policy and trust enforcement point at redemption and initialization time.

## Errors

Suggested status classes:

- `404` endpoint not found
- `406` unsupported `Accept`
- `5xx` transient Service Provider failure
