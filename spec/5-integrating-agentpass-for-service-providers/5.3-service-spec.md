# 5.3. AgentPass Service Provider Spec

## Overview

The AgentPass Service Provider Spec defines how a Service Provider validates and enforces delegated authority.

## Token Validation

A Service Provider MUST:

1. Validate token signature
2. Validate issuer trust
3. Validate expiration
4. Enforce scope constraints

If confirmation binding is present, the Service Provider MUST validate proof-of-possession.

## Effective Principal Construction

The Service Provider MUST construct an Effective Principal composed of:

- User
- Agent
- Delegated scope

The Service Provider MUST NOT treat the agent as equivalent to the User unless explicitly granted.

## Audit and Attribution

Service Providers SHOULD:

- Log both User and Agent identifiers
- Surface delegated activity distinctly
- Support activity callbacks to AgentPass where feasible

For AgentPass Bearer Tokens, User and Agent identifiers are carried in bearer token claims as defined in:

- [`5.5-agentpass-bearer-tokens.md`](5.5-agentpass-bearer-tokens.md)

## Deployment Resolution

Service Provider MUST provide a deployment-resolution endpoint that accepts User email and returns acceptable AgentPass deployment(s):

- a single authoritative deployment when authoritative discovery succeeds, or
- a set of explicitly trusted Federated AgentPass deployments when authoritative discovery is unavailable (and only from deployments explicitly defined in Service Provider Configuration (`trust.trusted_federated_deployments`)).

If authoritative discovery succeeds, Service Provider MUST NOT return federated options.

If a User email domain has an Authoritative AgentPass, Service Provider MUST only authorize agents delegated by that Authoritative AgentPass.
