# 5.5. Support AgentPass Bearer Tokens

Service Providers validate bearer tokens issued by AgentPass.

For bearer-token flows, Service Providers MUST expose and enforce the same deployment-resolution behavior defined for browser sessions (the Deployment Resolution Endpoint).

## Bearer Token Profile (Normative)

For AgentPass Bearer Token flows, `access_token` values issued by AgentPass MUST be JWTs secured as JWS (signed JWTs).

Service Providers MUST be able to validate token signatures using the issuing deployment's published keys (JWKS), obtained from the deployment configuration `jwks_uri` (Section `3.3`).

### Required Claims

An AgentPass Bearer Token MUST include:

- `iss` (string): the token issuer. Service Provider MUST validate this equals the deployment `issuer` from `GET {agentpass_configuration_url}` and MUST enforce authoritative vs federated precedence (Section `5.6.5`).
- `aud` (string or array of strings): the intended recipient. Service Provider MUST validate this matches the Service Provider (see definition of Audience (`aud`) in `../2-notational-conventions/2.1-definitions.md`).
- `exp` (number): expiration time.
- `iat` (number): issuance time.
- `sub` (string): the delegated User identifier. This MUST equal the Runtime-provided `request.user.sub` value from the Request Create Endpoint (`POST {agentpass_request_create_url}`).
- `agent` (object): agent identity context. This MUST include:
  - `agent.id` (string): the delegated Agent identifier. This MUST equal the Runtime-provided `request.agent.id` value from the Request Create Endpoint (`POST {agentpass_request_create_url}`).
- `scope` (string or array of strings): the delegated permission boundary.

An AgentPass Bearer Token MAY include:

- `cnf` (object): confirmation for holder binding (see `cnf` definition in `../2-notational-conventions/2.1-definitions.md`). If `cnf` is present, Service Provider MUST require proof-of-possession.

Runtimes MUST treat `access_token` as opaque. Service Providers enforce authorization based on validated token claims and local policy.

If `trust_mode = "federated"` for the User email domain (derived from `user.email`), Service Provider MUST validate that issuer is in the explicit allowlist of trusted Federated AgentPass deployments declared in the Service Provider Configuration (`trust.trusted_federated_deployments`).

If `trust_mode = "authoritative"` for the User email domain (derived from `user.email`), Service Provider MUST reject federated issuers.

If an Authoritative AgentPass is configured for the User, Service Provider MUST only authorize agents delegated by that Authoritative AgentPass.

Service Providers MUST validate:
- signature (JWKS from deployment configuration `jwks_uri`)
- trusted `iss` (and enforce authoritative precedence)
- `aud` matches the Service Provider
- `exp`
- scope/capabilities
- proof-of-possession if `cnf` is present

Service Providers MUST log User and Agent identifiers from validated bearer token claims (`sub` and `agent.id`).

## Decoded JWT Claims Example (Non-Normative)

```json
{
  "iss": "https://agentpass.example.com",
  "aud": "https://api.example.com",
  "iat": 1771041600,
  "exp": 1771042200,
  "sub": "user:colin",
  "agent": { "id": "agent:build-bot-7f2c" },
  "scope": ["tickets:read", "tickets:comment"]
}
```
