# 4.1. Initialize a AgentPass Browser Session

This section defines the normative Runtime flow for AgentPass Browser Sessions.

## Flow Context

This flow applies when a Runtime needs an agent-attributed browser session at a Service Provider.

The flow uses:

- authority request endpoints from Section `3.6`
- Service Provider authority-resolution and bootstrap endpoints from Section `5.2` and `5.4`

## Step 1: Gather Required Inputs

Before resolving authority, Runtime MUST have:

- `delegating_principal.email`
- `agent_principal.id`
- Service Provider origin (audience)

Before creating the authority request, Runtime MUST also have:

- `delegating_principal.sub` (stable subject identifier)
- `delegating_principal_domain` (from Service Provider authority-resolution response)

Runtime MAY include:

- destination URL intent
- runtime confirmation key material (`cnf`)

## Step 2: Discover Service Provider

Runtime MUST perform Service Provider discovery:

1. DNS query `_agentpass-service.{service_host}` TXT to resolve `service_e`.
2. `GET {service_e}` to fetch Service Provider metadata.

Discovery rules are defined in:

- [`../5-integrating-agentpass-for-service-providers/5.4-endpoints/5.4.1-discovery.md`](../5-integrating-agentpass-for-service-providers/5.4-endpoints/5.4.1-discovery.md)

## Step 3: Resolve Authority Through Service Provider

Runtime MUST call:

- `POST {service_e}/resolve-authority`

Request MUST include:

- `request.delegating_principal_email`

Response MUST include:

- `response.delegating_principal_domain`
- `response.trust_mode`
- `response.resolution`

If `response.trust_mode = "authoritative"`:

- Runtime MUST use `response.authoritative_authority` as selected authority.
- Runtime MUST NOT choose a federated authority.
- Runtime MUST treat this as authoritative configuration for the Delegating Principal.

If `response.trust_mode = "federated"`:

- Runtime MUST choose one authority from `response.trusted_federated_authorities`.
- Runtime MUST NOT use an authority outside that list.

Runtime SHOULD verify `response.resolution` is consistent with `response.trust_mode`:

- `authoritative` mode -> `resolution = "authoritative"`
- `federated` mode -> `resolution = "federated_options"`

Runtime MUST NOT perform direct authoritative discovery as part of this flow.

## Step 4: Perform Trust Compatibility Pre-check

Runtime SHOULD verify selected authority against Service Provider trust metadata when present.

- If `trust_mode = "authoritative"` and `service.trust.accepted_authorities` is present, selected issuer MUST be in that set.
- If `trust_mode = "federated"` and `service.trust.accepted_federated_authorities` is present, selected issuer MUST be in that set.
- If these fields are absent, Runtime MAY proceed, but Service Provider remains final enforcement point.

## Step 5: Create Browser Sessions Request

Runtime MUST call:

- `POST {authority_e}/requests`

Request body MUST conform to:

- [`../3-operating-agentpass/3-schemas/request-create.schema.json`](../3-operating-agentpass/3-schemas/request-create.schema.json)

For AgentPass Browser Sessions, Runtime MUST set:

- `type = "service_signin_bootstrap"`
- `service.origin`
- `delegating_principal.sub`
- `delegating_principal_domain` (from Service Provider resolution response)
- `agent_principal.id`

If Runtime requests holder binding, it SHOULD include `runtime.cnf`.

If Runtime has a user-intended destination, it SHOULD include `intent.destination_url`.

## Step 6: Poll for Approval Status

Authority MAY require human approval and asynchronous processing.

Runtime MUST poll:

- `GET {authority_e}/requests/{id}`

Status response MUST conform to:

- [`../3-operating-agentpass/3-schemas/request-status.schema.json`](../3-operating-agentpass/3-schemas/request-status.schema.json)

### Status lifecycle

Runtime MUST handle these status values:

- non-terminal: `pending`
- terminal: `approved`, `denied`, `expired`, `canceled`

Allowed transition model:

- `pending -> approved | denied | expired | canceled`
- terminal states do not transition further

### Polling behavior

While status is `pending`, Runtime MUST continue polling until terminal state.

- If `poll_after_ms` is present, Runtime SHOULD honor it.
- If absent, Runtime SHOULD back off between polls and MUST avoid tight-loop polling.

### Approved result

When status is `approved`, response MUST include an `artifact` suitable for AgentPass Browser Sessions redemption.

If approved status does not include redeemable artifact data, Runtime MUST treat response as protocol-invalid and fail flow.

## Step 7: Redeem Artifact at Service Provider

After approval, Runtime MUST call:

- `POST {service_e}/init`

Request/response MUST conform to:

- [`../3-operating-agentpass/3-schemas/service-init.schema.json`](../3-operating-agentpass/3-schemas/service-init.schema.json)

Runtime MUST include:

- `request.artifact` from approved authority response

Runtime SHOULD include:

- `request.runtime_proof` when proof-of-possession is required

Runtime MUST include `request.delegating_principal_domain` when needed for precedence enforcement, including when artifact content is opaque to Service Provider.

## Step 8: Initialize Browser Session

If `POST {service_e}/init` succeeds, Runtime receives `initialization_url`.

Runtime MUST:

- treat `initialization_url` as single-use and short-lived
- load it directly in emulated browser
- avoid prefetching, retries, or secondary sharing of URL

If browser initialization fails after URL issuance, Runtime MUST create a new request instead of reusing same URL.

## Step 9: Handle Terminal Errors

Runtime MUST treat these outcomes as terminal failures for current request:

- authority status is `denied`, `expired`, or `canceled`
- Service Provider redemption rejects artifact (`4xx`) or replay semantics are triggered
- initialization URL is expired or already consumed

After terminal failure, Runtime MAY start a new request with fresh identifiers.

## Step 10: Apply Security Requirements

Runtime MUST:

- protect artifact and token material from logs and telemetry
- present artifacts only to intended Service Provider origin
- bind requests to holder key material when required by authority/Service Provider policy
- preserve delegating + agent identity attribution for downstream audit
