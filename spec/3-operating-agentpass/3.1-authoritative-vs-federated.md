# 3.1. Authoritative vs Federated

Each AgentPass deployment operates in one of two trust modes:

- `authoritative`
- `federated`

An AgentPass deployment MUST declare its `trust_mode` in its configuration (`GET {agentpass_configuration_url}`).

## 3.1.1. Authoritative Trust Mode

`trust_mode = "authoritative"` (an **authoritative AgentPass deployment**) is intended for organizations that need enterprise control over how their members delegate to agents, including policy enforcement, approval control, and auditable attribution.

In authoritative mode, the AgentPass deployment discovered for the User email domain (derived from `user.email`) via `_agentpass.{user_email_domain}` is the source of truth for delegation policy and authorization for that domain.

For interoperable implementations, authoritative mode is bootstrapped via DNS TXT discovery (Section `3.2`) followed by configuration fetch (`GET {agentpass_configuration_url}`).

## 3.1.2. Federated Trust Mode

`trust_mode = "federated"` (a **federated AgentPass deployment**) is used when the Service Provider does not have a usable authoritative deployment for the User and elects to trust one or more federated deployments by policy.

Federated providers are high-trust dependencies and should be explicitly selected by Service Provider policy.

## 3.1.3. Precedence Rule (Normative)

The precedence rule is enforced by the Service Provider when accepting artifacts/tokens.

If authoritative DNS TXT discovery succeeds for the User email domain (derived from `user.email`), the Service Provider MUST attempt to use the discovered authoritative deployment for that User.

If authoritative discovery returns `none`, the Service Provider MUST reject delegation for that User and MUST NOT fall back to a federated deployment.

If DNS discovery returns a URL but anything after DNS discovery fails (for example, HTTPS configuration fetch or validation), the Service Provider MUST reject delegation for that User and MUST NOT fall back to a federated deployment.

When the Service Provider operates in authoritative mode for a User, it MUST accept only artifacts/tokens whose `iss` matches the authoritative deployment `issuer` obtained from configuration for the discovered endpoint, and whose signatures validate using the authoritative deployment's published keys.

When the Service Provider operates in federated mode, it MUST accept only artifacts/tokens issued by federated deployments it explicitly selected by policy (for example, deployments returned as trusted options for that User by the Service Provider deployment-resolution endpoint).

Discovery and configuration are defined in Sections `3.2` and `3.3`:

- [`3.2-discovery.md`](3.2-discovery.md)
- [`3.3-configuration.md`](3.3-configuration.md)
