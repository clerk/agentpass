# 5.2. Support AgentPass Browser Sessions

This section defines Service Provider behavior for AgentPass Browser Sessions.

## Endpoints

Service Providers expose a base endpoint `e` discovered via DNS and metadata.

- `GET {e}` returns Service Provider metadata.
- `POST {e}/resolve-deployments` resolves acceptable AgentPass deployment(s) for a delegating principal email.
- `POST {e}/init` redeems a bootstrap artifact.
- `GET {e}/initialize/{token}` establishes browser session and redirects.

## POST {e}/resolve-deployments Contract

Request/response MUST conform to:

- [`../3-operating-agentpass/3-schemas/service-authority-resolution.schema.json`](../3-operating-agentpass/3-schemas/service-authority-resolution.schema.json)

Service Provider MUST:

- accept `request.delegating_principal_email`
- derive `delegating_principal_domain` from email domain
- perform authoritative discovery for that domain

If authoritative discovery succeeds, Service Provider MUST return:

- `response.authoritative_deployment`

If authoritative discovery does not resolve, Service Provider MUST return:

- `response.trusted_federated_deployments`

Service Provider MUST NOT return federated options when authoritative discovery succeeds.

## POST {e}/init Request Contract

Request/response MUST conform to:

- [`../3-operating-agentpass/3-schemas/service-init.schema.json`](../3-operating-agentpass/3-schemas/service-init.schema.json)

### Required request fields

Service Provider MUST require:

- `request.artifact.type`
- `request.artifact.value`

Service Provider SHOULD accept:

- `request.runtime_proof`
- `request.delegating_principal_domain`

### Validation algorithm

Service Provider MUST perform all checks before issuing `initialization_url`:

1. Protocol validation (required fields, parseability).
2. Artifact integrity validation (signature or equivalent authenticity mechanism).
3. Artifact freshness (`exp`/TTL) and single-use status.
4. Audience validation: artifact MUST target this Service Provider origin.
5. Issuer trust validation against Service Provider trust policy.
6. Authoritative precedence enforcement for `delegating_principal_domain`.
7. Runtime proof validation when artifact or policy requires holder binding.
8. Destination validation against Service Provider allow-policy.

If any check fails, Service Provider MUST reject redemption and MUST NOT mint initialization state.

### Trust-Mode Precedence Enforcement

Service Providers MUST enforce Section `3.1` trust-mode precedence:

- If authoritative discovery for `delegating_principal_domain` succeeds, Service Provider MUST accept only artifacts issued by that authoritative provider.
- If authoritative discovery does not resolve, Service Provider MAY accept trusted federated issuers.
- If Service Provider previously returned `trusted_federated_deployments` for this sign-in, Service Provider MUST enforce issuer membership in that returned set.

If an authoritative AgentPass deployment is configured for the Delegating Principal, Service Provider MUST only authorize agents delegated by that authoritative deployment.

Reference:

- [`../3-operating-agentpass/3.1-authoritative-vs-federated.md`](../3-operating-agentpass/3.1-authoritative-vs-federated.md)
- [`../3-operating-agentpass/3.5-federated-identity-providers.md`](../3-operating-agentpass/3.5-federated-identity-providers.md)

If artifact does not carry domain information directly, Service Provider SHOULD use `request.delegating_principal_domain`.

## POST {e}/init Response Contract

On success, Service Provider MUST return:

- `response.initialization_url`

Service Provider SHOULD return:

- `response.expires_at`
- `response.one_time = true`

### Initialization URL requirements

`initialization_url` MUST be:

- single-use
- short-lived
- bound to the approved artifact redemption context

Service Provider MUST generate initialization state atomically with single-use marking to prevent replay races.

## GET {e}/initialize/{token}

When handling initialization token, Service Provider MUST:

1. Verify token exists, is unexpired, and is unused.
2. Consume token atomically as used before session finalization.
3. Establish an agent-attributed session for the approved delegating principal.
4. Enforce approved destination constraints before redirect.

Service Provider MUST reject:

- unknown token
- expired token
- already-consumed token

Service Provider MUST NOT allow re-use after first successful or failed terminal consumption attempt.

## Error Handling

For redemption and initialization endpoints, Service Provider SHOULD return structured JSON errors.

At minimum, error responses SHOULD include:

- machine-readable `code`
- human-readable `message`

Suggested classes:

- `400` malformed request
- `401` invalid runtime proof (when applicable)
- `403` trust or policy rejection
- `409` single-use conflict / replay
- `410` expired or consumed initialization token
- `422` semantically invalid artifact

## Browser and Session Security

Service Provider MUST:

- issue session cookies with secure attributes appropriate for browser security
- prevent open redirect behavior
- set response controls to avoid caching of one-time initialization material

Service Provider SHOULD ensure initialization endpoints are resistant to CSRF and token leakage via referrer or logs.

## Audit and Attribution

Service Providers SHOULD log:

- request identifier
- delegating principal identifier
- agent principal identifier
- authority issuer
- runtime identifier (when available)
- redemption/initialization outcome
