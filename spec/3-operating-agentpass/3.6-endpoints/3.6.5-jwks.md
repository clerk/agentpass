# 3.6.5. JWKS

Authority exposes signing keys for bearer tokens and/or verifiable artifacts.

## Discovery

Clients obtain `jwks_uri` from deployment metadata:

- `GET {e}` -> `crypto.jwks_uri`

Reference:

- [`3.6.1-get-authority-metadata.md`](3.6.1-get-authority-metadata.md)

## Endpoint Requirements

If an authority issues signed bearer tokens or signed artifacts, it MUST expose JWKS over HTTPS.

JWKS endpoint MUST return a JSON Web Key Set document (`application/json`) containing verification keys used by current and recently-issued credentials.

Each key SHOULD include:

- `kid`
- `kty`
- algorithm-compatible key material

## Validation Requirements

Clients and Service Providers MUST select keys by `kid` when present.

If `kid` is absent, verifiers MAY try compatible keys but MUST fail closed when signature validation is ambiguous.

## Rotation and Caching

Authorities SHOULD support key rotation with overlap, so previously-issued credentials remain verifiable until expiration.

Verifiers SHOULD cache JWKS and re-fetch on:

- unknown `kid`
- signature validation failure with otherwise-valid credential context
- cache expiry

## Errors

Suggested status classes:

- `404` JWKS unavailable
- `429` rate limited
- `5xx` transient failure
