# 3.3. AgentPass Core Specification

## Overview

AgentPass defines a governance model for delegated agent authority.

AgentPass is the authority layer responsible for:

- Establishing delegation between a human and an agent
- Issuing delegated authority tokens
- Continuously evaluating policy
- Supporting revocation and oversight
- Providing visibility into agent activity

## Roles

### Delegating Principal
A human identity that authorizes an agent. See `../2-notational-conventions/2.1-definitions.md`.

### Agent Principal
A software identity that operates on behalf of a Delegating Principal. See `../2-notational-conventions/2.1-definitions.md`.

### AgentPass
The governing authority system responsible for delegation lifecycle and policy enforcement. See `../2-notational-conventions/2.1-definitions.md`.

### AgentPass Token
A cryptographically signed representation of delegated authority.

## Delegation Model

AgentPass models:

- Delegating Principal
- Agent Principal
- Effective Principal (Human + Agent context)

AgentPass MAY enforce continuous policy evaluation and real-time governance.

## Token Requirements

An AgentPass Token MUST:

- Identify the Delegating Principal
- Identify the Agent Principal
- Specify scope or capabilities
- Include issuance and expiration times
- Be cryptographically signed

## Continuous Governance

AgentPass MAY:

- Require token refresh per action
- Provide introspection endpoints
- Emit activity events
- Require step-up approval
- Support revocation at any time

## Trust Modes

AgentPass deployments operate in one of two trust modes:

- `authoritative`
- `federated`

Multiple AgentPass instances can exist on the internet; each instance may be authoritative for its managed principals or federated for Service Provider allowlist use.

Normative trust-mode and precedence rules are defined in:

- [`3.1-authoritative-vs-federated.md`](3.1-authoritative-vs-federated.md)
- [`3.5-federated-identity-providers.md`](3.5-federated-identity-providers.md)

### Delegating Principal Domain

For flows that require authoritative discovery and federated precedence enforcement, requests and artifacts/tokens MUST include `delegating_principal_domain`. See `../2-notational-conventions/2.1-definitions.md` for canonical term definition. Service Provider deployment-resolution flows require delegating principal email as input.
